\section{Problem Setting}

The automated testing framework alleviates the user's efforts from providing numerous unit test cases by using random test generators to produce these test cases automatically. 
As a consequence, the performance of the automated testing frameworks are highly depends their test generation procedure\cite{?}. Precisely, the effectiveness (i.e., if potential bugs can be detected by the automated testing) is depends on the \emph{coverage completeness} of corresponding test generators (i.e., if all possible test inputs can be produced); on the other hand, the efficiency is controlled by the likelihood that test generators produce useless, duplicates, or vacuous erroneous test inputs that directly violate \emph{safety} properties. While the automated test framework enables a push-button bug detection; the ``last mile'' problem is still critical, that is ``who provides the desirable test generators'' . Implementing a proper test generator that guarantee coverage completeness and safety is hard because of its non-deterministic natural. Moreover, this problem is more complicated for distributed systems testing tasks, where automated testing needs to target \emph{open system}, e.g., the system involves \emph{interactions} between test generators (e.g., clients) and code under test (e.g., servers). Consider a bank server that maintains the balances of accounts and provides query and withdraw services to clients. In order to test the implementation of bank server, a desirable test generator needs to simulate the behaviours of expect clients -- send multiple query and withdraw request messages to servers.
Different from the standard ``one-shot'' style testing, where test generator would never involve testing after it passes the test inputs to the code under test; open system needs ``multi-shot'' style testing, where the test inputs, or we call \emph{request messages}, produced test generator may depend on the previous \emph{response messages} from the servers under test.

This open system increase the difficulty of providing safety and coverage completeness guarantee for test generator implementation. In bank server example, the clients should not withdraw amount that is greater than its current balance, where current balance comes by the query response message from bank server. In fact, the property of test generator implementation may involve temporal dependency on request and response messages that comes from black-box code under test and totally out of control from test generator itself. In this sense, it is even hard to define a fixable coverage completeness property for test generators.

On the other hand, the open system also makes the test framework harder to control the searching space of test cases. For normal test input generators, users typically would like to control the size of generated test inputs (e.g., the length of lists, the depth of trees). On the other hand, the ``size'' of test cases in open system setting may be interfered by code under test. In bank server example above, users may expect the test generator finish $5$ \emph{success} withdraw actions within sending $20$ messages. Note that, some withdraw request may fail or be missing; thus, the test generators may need to repeat failed requests. However, the times of repeating is depends on the remaining message budget and the number of succeed withdraw actions, which is controlled by code under test. Thus, A more complicate dynamic \emph{sampling} strategy within the searching space is expected.

To address the challenges above, we design a trace-based test generator compiler that compiles user-defined logic specification into executable test generators. The workflow is shown in \autoref{fig:workflow}. The main contributions are
\begin{enumerate}
    \item Decomposed the user inputs as two parts, \emph{quantified} specification and \emph{configurations}. The specification is the property that compiled test generator expected to hold, while the configuration parameterize the searching space to control the complexity of automated testing. For example, the specification shown in \autoref{fig:workflow} is quantified with key $x$ that is belongs to in an uninterpreted key type ($\Pi key{:}\mathcal{T}, \forall x{:}key$); this uninterpreted key type will be instantiated by configuration as a concrete type that has finite inhabitants ($key \leftarrow [1;2;3]$) during testing. Moreover, we also allowing different \emph{sampling strategies} for test input generation, includes length-based, state-based, and transition based strategies. As shown in \autoref{fig:workflow}, the configuration indicates the length of traces cannot be greater than $10$, which helps the a length-based sampling strategy to produce requests messages.
    The compiled test generators can be reusable under different configurations, which facilitates to build multiple test scenarios from different configurations.
    \item There are two kinds of specifications, \emph{local specification} and \emph{global specification}. The global specification constrains traces involve both request message generated by test generators and response message from servers (e.g., specification shown in \autoref{fig:workflow}). On the other hand, the local specification only specifies the traces involves request messages generated by test generators after filtering. Unlike the global specification indicates the safety guarantee and interpreted as overapproximation; the local specification will be interpreted as both over- and under-approximation, which means it is both safety and coverage complete property.
    \item We uses (quantified) symbolic regular expression (SRE) and symbolic regular expression with memory (SREM) are intermediate language, which can be translated into (quantified) symbolic finite automata (SFA) and symbolic register automata (SRA). Moreover, we use a quantified symbolic linear temporal logic on finite sequence (LTLf) as our surface language to help users write specifications easier. The register automata is only used in global specification that helps us store the response from the server.
    \item We design a new compilation procedure to translate quantified SFA and SRA into state machines in P language. We guarantee this translation is sound and partial complete (complete with respect to local specifications).
\end{enumerate}

\section{Case Studies} %
\label{sec:case}

This section illustrates the expressiveness of our bunched type system
via three real-world case studies.

\paragraph{May/Must Analysis} %
\label{subsec:may-must}
\begin{figure}

?$\vdash$? let f (x: int) (y: int) =

    else x + 1
    : ?$\Code{x}{:}\ \urt{\Int}{\vnu = 0} \ctsarr \Code{y}{:}\urt{\Int}{0 \leq \vnu \leq 10} \ctsarr$?

Prior work has considered program analyses that reason about may- and
must-properties within the same framework~\cite{BVR+08,GN+10}. Like
our type system, these analyses are compositional because callee
summaries inform caller deductions. Figure~\ref{fig:may-must-analysis}
adapts examples from \citet{GN+10}, restated in our
core language with context types assigned to both the caller
$\Code{f}$ and the callee $\Code{g}$. In our encoding, demonic types
correspond to may-analysis summaries (safety), while angelic types
correspond to must-analysis summaries (reachability).

Figure~\ref{fig:may-must-analysis}~(a) illustrates how a may-analysis
result on $\Code{g}$ enables a must-analysis result on
$\Code{f}$. $\Code{f}$'s type asserts that it must return 1, given
that $\Code{x}$ can be 0 and $\Code{y}$ can be any value in
[0,10]. The true branch ($\Code{y} > 10$) would require
$\Code{g}\;\Code{x} = 0$, but $\Code{g}$'s demonic type precludes
this.  The true branch will therefore never provide a valid witness
for $\Code{f}$'s reachability guarantee. Focusing on the false branch
instead by angelically choosing $\Code{y} \in [0, 10]$, reveals that
calling $f$ with $\Code{x = 0}$ causes it to return
$\Code{x} + 1 = 1$, and justifying the target type.

Figure~\ref{fig:may-must-analysis}~(b) illustrates how a must-analysis
result on $\Code{g}$ enables a \emph{tight} may-analysis result on
$\Code{f}$ whose type is a safety specification asserting that it
always returns either 2 or 5.  The true branch ($\Code{a} > 3$) always
returns 5, and the false branch ($\Code{a} \leq 3$) always returns 2.
Function \Code{g}'s safety specification ($\ort{\Int}{\vnu > 0}$)
establishes that $\Code{a} > 0$, ensuring neither branch produces an
out-of-range value. However, with only this type, a may analysis
cannot assert that both 2 and 5 are \emph{definitively} part of
$\Code{f}$'s behavior.  With $\Code{g}$'s must type ($\urt{\Int}{\vnu
  > 0}$), however, such an analysis can conclude that $\Code{a}$ can
reach values that are both $\leq 3$ and $>3$, guaranteeing that both
branches are definitely reachable. This makes $\Code{f}$'s result type
provably tight: the must information certifies that neither 2 nor 5 is
spurious.


\paragraph{Coverage Completeness}%
\label{subsec:coverage}
\begin{wrapfigure}{r}{.55\textwidth}
  \vspace*{-.2in}
\begin{minted}[xleftmargin=15pt, numbersep=4pt, linenos = true, fontsize = \footnotesize, escapeinside=??]{OCaml}
let rec bst_gen (r: nat) (lo: nat)
              (range: nat -> nat -> nat): nat tree =
if r <= 0 then Leaf
else Leaf ?$\oplus$?
  let r' = range (1, r) in
  let left  = bst_gen (r' - 1) lo range in
  let right = bst_gen (r - r') (lo + r') range in
  Node (lo + r') left right
\end{minted}
\caption{A BST generator adapted from~\cite{CoverageType}.}
  \label{fig:bst}
\vspace*{-.1in}
\end{wrapfigure}
The coverage type system of~\citet{CoverageType} verifies that a test
generator produces every desirable input.  Given the BST generator
shown in Figure~\ref{fig:bst}, the following specification asserts
that for any valid range generator, \texttt{bst\_gen} produces
\emph{every} BST whose keys lie within
$(\Code{lo}, \Code{lo}+r]$:\footnote{In the type for
  $\Code{bst\_gen}$, $\Code{mem}(\vnu,x)$ is a \emph{method
    predicate}, i.e., an uninterpreted function used to encode
  properties of algebraic data types like trees, in this case a tree
  membership property.}
\begin{align*}
  \mykw{val}\;\Code{bst\_gen} : \ctpers(&
    \Code{r}{:}\ort{\Nat}{\top} \ctsarr \Code{lo}{:}\ort{\Nat}{\top} \ctsarr
  \\&
    \Code{range}{:}\ctpers (x{:}\ort{\Nat}{\top} \ctsarr
      y{:}\ort{\Nat}{x < \vnu} \ctsarr
      \urt{\Nat}{x < \vnu < y})
    \ctsarr
  \\ & \quad
    \urt{\Nat\;\Code{tree}}{\Code{bst}(\vnu) \land
      \forall x.\, \Code{mem}(\vnu, x) \iff
      (\Code{lo} < x \leq \Code{lo}+r)})
\end{align*}
In our setting, input generators always introduce fresh angelic
capabilities at each invocation, independently of the surrounding
context, for \emph{arbitrary} input parameters. Input generators thus
have context types of the following shape:
\begin{align*}
    \overbrace{\ctpers (\quad \underbrace{\overline{x{:}\ort{b_x}{\phi_x}\ctsarr}}_{\texttt{demonic parameters}} \quad \underbrace{\urt{b}{\phi}}_{\texttt{angelic result}} \quad )}^{\texttt{duplicatable}}
\end{align*}\noindent
The \Code{range} parameter of \Code{bst\_gen}, for example, is
generator is persistent and has demonic parameter types
($\ort{\Nat}{\top}$ and $\ort{\Nat}{x < \vnu}$) and an angelic result
type ($\urt{\Nat}{x < \vnu < y}$).  The demonic parameters allow
successive calls to \Code{range} with different arguments to be typed
with independent result capabilities --- the generator's output at
each recursive call is disjoint from its outputs at other calls.  The
BST generator follows the same pattern: its demonic parameter types
allow any valid size $\Code{r}$ and lower bound $\Code{lo}$, and its
angelic result type asserts that every valid BST is reachable by some
sequence of generator choices.

\paragraph{Incorrectness Reasoning} %
\label{subsec:disprove}
\begin{figure}[h!]
  \centering
\begin{minted}[xleftmargin=5pt, numbersep=4pt, linenos = true, fontsize = \footnotesize, escapeinside=??]{OCaml}
    let rec merge (l1: nat list) (l2: nat list) : nat list =
        match l1, l2 with
        | l1, [] -> l1
        | [], l2 -> l2
        | h1 :: t1, h2 :: t2 ->
            let t = merge t1 t2 in (* t contains elements in t1 *)
            if h1 < h2 then h1 :: h2 :: t (* h2 may be greater than elements in t1 *)
            else if h1 > h2 then h2 :: h1 :: t
            else h1 :: t
\end{minted}
\caption{A function that merges two sorted lists. If either of
the input lists are empty, \Code{merge} returns the other (lines
$3-4$). Otherwise, the function merges the tails of the two inputs and appends the two top elements ahead of the merged result. }
\label{fig:merge}
\end{figure}
When a refinement type system fails to prove a safety property, the
target program is not necessarily incorrect: may-style
(overapproximate) reasoning offers no guarantee that a presumed
erroneous behavior is actually reachable. Recent
work~\cite{RefinementTypeRefutations, RefinementTypeSymbolicExecution}
addresses this limitation by using symbolic execution to search for
counterexamples; our context type system can establish such
counterexamples directly.  Consider the $\Code{merge}$ function shown
in Figure~\ref{fig:merge}. When used in a mergesort procedure, this
function is intended to merge two sorted lists and produce a sorted
output list:
\begin{align*}
  \vdash \Code{merge}~:~\Code{l1}{:}\ort{\List[\Code{nat}]}{\top}
  \ctsarr \Code{l2}{:}\ort{\List[\Code{nat}]}{\top} \ctsarr
  \ort{\List[\Code{nat}]}{\I{sorted}(\vnu)} \quad \judgfail
\end{align*}\noindent
Verification fails nonetheless. The base cases (lines $3$--$4$)
type-check, so the failure arises in the recursive case, but the
refinement type system offers little guidance on the underlying fault,
since it cannot identify which execution branch leads to the unsorted
output.  A symbolic execution engine might identify the following
concrete counterexample:
\begin{align*}
    \texttt{Counterexample:}&\quad  \text{ when } \Code{l1} = [1;2] \text{ and } \Code{l2} = [3;4] \text{ then } \mstep{\Code{merge}\;[1;2]\;[3;4]}{[1;3;2;4]}
\end{align*}\noindent
The failure stems from the assumption that, after comparing
$\Code{h1}$ and $\Code{h2}$, $\Code{h2}$ is always the second-smallest
element (line $7$).  Even when $\Code{h1} < \Code{h2}$, however,
$\Code{h2}$ may still be larger than the next element of $\Code{l1}$,
breaking sortedness.  Rather than appealing to a separate symbolic
execution procedure to discover this counterexample, we can directly
prove that \Code{merge} operates incorrectly, by type-checking it with
the following type:
\begin{align*}
    &\Code{l1}{:}\urt{\List[\Code{nat}]}{\top} \ctwand \Code{l2}{:}\urt{\List[\Code{nat}]}{\top} \ctwand \ort{\List[\Code{nat}]}{\neg \I{sorted}(\vnu)} \ctoplus \ort{\List[\Code{nat}]}{\top}
\end{align*}\noindent
This type follows the ``affine reachability'' style of
Section~\ref{sec:overview}: the result type is a sum of the top type
$\ort{\List[\Code{nat}]}{\top}$ and the negation of the original
safety property, $\ort{\List[\Code{nat}]}{\neg \I{sorted}(\vnu)}$,
that captures the erroneous behavior.  The parameter types are given
\emph{disjoint angelic types}, indicating that $\Code{l1}$ and
$\Code{l2}$ may independently range over arbitrary sorted lists.
Without this disjointness, a carefully crafted context can hide the
bug, as shown below. Here $\Code{l1}$ and $\Code{l2}$ each
\begin{wrapfigure}{l}{.4\textwidth}
  \begin{center}
    \vspace*{-.1in}
\begin{minted}[xleftmargin=5pt, numbersep=4pt, linenos = true, fontsize = \footnotesize, escapeinside=??]{OCaml}
  let l1 = sorted_list_gen () in
  let l2 = match l1 with
           | [] -> sorted_list_gen ()
           | _ -> [] in
  merge l1 l2
\end{minted}
\end{center}
\end{wrapfigure}
range over all sorted lists, given by $\Code{sorted\_list\_gen}$, yet
they cannot both be non-empty simultaneously.  Under this environment,
the bug in \Code{merge} cannot be triggered, since the case where both inputs
are non-empty (line $5$) is never reached.

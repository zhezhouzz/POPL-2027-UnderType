\section{Introduction}\label{sec:intro}

Type-based reasoning has become a standard approach to functional program verification.
Refinement types are a prominent example: they support overapproximate safety reasoning by embedding logical safety properties directly into types.
For a typing judgment $\Gamma \vdash e : \tau$,
the result of $e$ may lie in the set described by the type $\tau$ under that context.
Not every behavior admitted by $\tau$ needs to be realized by $e$, so $\tau$ is an overapproximation of $e$'s behavior.
This relation must hold for all \emph{environments} admitted by type context $\Gamma$, yielding the usual safety guarantee: a well-typed program never goes wrong under all possible environments. We can interpret this in another way: the type context $\Gamma$ is \emph{demonic}, choosing the worst possible context adversarially. In order to defend against the demonic context, we should make sure that $e$ is consistent with type $\tau$ under all possible environments that the demonic context can choose.

Dually, we can adopt an underapproximate interpretation, where $\tau$ gives a lower bound of $e$'s behavior under an angelic-style context $\Gamma$. Consider the following example:
\begin{align*}
    &x{:}\ort{\Nat}{\vnu > 0}, y{:}\ort{\Nat}{\vnu > 0} \vdash x - y : \ort{\Nat}{\vnu > 0} \judgfail
    \\&x{:}\urt{\Nat}{\vnu > 0}, y{:}\urt{\Nat}{\vnu > 0} \vdash x - y : \urt{\Nat}{\vnu > 0}
\end{align*}\noindent
where we use $\ort{b}{\phi}$ to express that a primitive type $b$ is refined with a qualifier $\phi(\vnu)$ in overapproximate and demonic style, and $\urt{b}{\phi}$ for underapproximate and angelic style. The two judgements have the same term and qualifier, but different interpretations. The first judgement asks a safety question:

\textbf{Overapproximation and Demonic Context:} \emph{If the type context can assign any positive number to $x$ and $y$ demonically, is the execution result of the term $x - y$ still positive?}

\noindent The answer is no, since a demonic type context can choose a bad environment $[x \mapsto 1, y \mapsto 1]$, which makes the execution result of the term $x - y$ equal to $0$. Alternatively, the second judgement asks a reachability question:

\textbf{Underapproximation and Angelic Context:} \emph{Do there exist desirable positive numbers that an angelic type context can assign to $x$ and $y$ such that every positive number must be a result of the term $x - y$?}

\noindent The answer is yes. Assume we want to reach a positive number $n$; then the angelic context can choose a desirable environment $[x \mapsto n + 1, y \mapsto 1]$. Note that the angelic context first knows the reachability goal and then chooses the environment, which enables ``reverse-style'' reasoning.


The power of this underapproximate interpretation and angelic context has been widely studied in program logic and proved useful for reachability reasoning. For example, Reverse Hoare Logic~\cite{ReverseHoareLogic} uses a similar ``reverse-style'' reasoning to prove reachability of imperative programs for randomized algorithms (e.g., all permutations of a list must be generated by a $\Code{shuffle}$ function). Another underapproximate-style program logic is Incorrectness Logic~\cite{OHP19}. It avoids overwhelming false positives in bug finding by using an angelic context (``angelic'' means finding hidden bugs luckily) to provide proper input states in the precondition that justify the erroneous behavior of a program. Later, Outcome Logic~\cite{ZDS23} and follow-up works in this style provide more unified and powerful reasoning for concurrent or probabilistic programs. However, the analogue in type-based verification for functional languages is not yet established. Although Coverage Type~\cite{CoverageType} interprets types as underapproximations of test-input-generator behavior and can prove coverage completeness (e.g., all binary search trees can be generated nondeterministically), none of these type-based works consider angelic context as shown in the previous example.

Enabling angelic context in type-based verification of functional languages is more challenging than in program logic. The context in a functional language can also provide function values demonically or angelically, where the functions themselves can have their own context. Consider the previous example in a curried-isomorphic form:
\begin{align*}
    &x{:}\urt{\Nat}{\vnu > 0} \vdash \lambda y.x - y : y{:}\urt{\Nat}{\vnu > 0} \sarr \urt{\Nat}{\vnu > 0} \tag{$\tau_f$}
\end{align*}\noindent
We highlight that there are two input contexts for a lambda function: a ``captured context'' $x{:}\urt{\Nat}{\vnu > 0}$ contains the variables referred to by the function $\lambda y.x - y$, while ``parameter context'' $y{:}\urt{\Nat}{\vnu > 0}$ is the context expected by this function during application. This complicated context dependency makes higher-order functions more difficult to reason about.
\begin{align*}
    x{:}\urt{\Nat}{\vnu > 0}, y{:}\urt{\Nat}{\vnu > 0} &\vdash  \lambda f. f\;x : f{:}\tau_f \sarr \urt{\Nat}{\vnu > 0} \judgfail
\end{align*}\noindent
As shown above, the higher-order function $\lambda f. f\;x$ takes a function $f$ that is assumed to have type $\tau_f$ discussed above and directly applies $f$ to $x$. At first glance, it should guarantee reachability of all positive numbers. However, when $f$ is the function $\lambda y.x - y$, it always returns $0$.
\begin{align*}
    &f_1 \doteq \lambda y.x - y \quad (\lambda f. f\;x)\;f_1 \;\hookrightarrow^*\; x - x \;\hookrightarrow^*\; 0
\end{align*}\noindent
The problem is that the ``captured context'' of function $\lambda y.x - y$ contains variable $x$, which creates a hidden dependency: parameter $y$ is expected not to alias with $x$. Moreover, a similar alias problem can happen with the ``result context'' of function application. Consider the following nested-application example:
\begin{align*}
    x{:}\urt{\Nat}{\vnu > 0}, y{:}\urt{\Nat}{\vnu > 0} \vdash \lambda f. f(f\;y) : f{:}\tau_f \sarr \urt{\Nat}{\vnu > 0} \judgfail
\end{align*}\noindent
Although the first application $f\;y$ works as expected, the second application $f(f\;y)$ uses the result of the previous application, which is problematic. Here is a counter-example:
\begin{align*}
    &f_2 \doteq \lambda y.\zif{y = 1}{x}{2} \quad (\lambda f. f\;(f\;y))\;f_2 \;\hookrightarrow^*\; f\;x \;\hookrightarrow^*\; \zif{x = 1}{x}{2}
\end{align*}\noindent
The function $\lambda y.\zif{y = 1}{x}{2}$ can have type $\tau_f$, since the parameter context can angelically assign value $1$ to $y$ to make the function return $x$. However, the result of the first application then depends on $x$, which makes the nested application fail to reach any value greater than $2$.

In general, context dependency is the core challenge for reasoning about angelic context in functional languages that enable higher-order functions. The design of type context should carefully handle three different contexts related to a function. That is, a function not only takes ``parameter context'' as input and produces ``result context'' as output, but can also have hidden ``captured context'' referred to by the function. For a curried type context $\overline{x_i:\tau_i}$ that can be decomposed into sub-contexts with binders $x_i$, it is important for a type system to track whether two variables are dependent or disjoint. Note that this complexity is not revealed in standard refinement type systems, which only involve demonic context. The intuition is that angelic context performs ``reverse-style'' reasoning that allows the environment of a function to remain ``undecided'' until the result of function application is known. However, this reverse reasoning conflicts with call-by-value semantics, which requires function application to happen after the argument value is decided.

In order to define a fundamental type theory that supports type context in both angelic and demonic style, we treat overapproximation (i.e., demonic style) and underapproximation (i.e., angelic style) as first-class modalities, and support fine-grained mixing in arbitrary ways. Moreover, to solve the context dependency issue, we treat context as \emph{resources} and introduce a \emph{resource algebra} for principled, compositional decomposition and combination of contextual pieces. This approach enables us to handle context disjointness in the same way as in bunched implication (e.g., separation logic) and bunched type systems. We then introduce the normal function type $\ctsarr$ and magic wand type $\ctwand$ in our \emph{context types} to distinguish whether the ``captured context'' is dependent on or disjoint from the ``parameter context''. For example,
\begin{align*}
    &x{:}\urt{\Nat}{\vnu > 0} \vdash \lambda y.x - y : y{:}\urt{\Nat}{\vnu > 0} \ctwand \urt{\Nat}{\vnu > 0}
\end{align*}\noindent
The magic wand above requires parameter $y$ to be independent of the current context (i.e., $x{:}\urt{\Nat}{\vnu > 0}$), which guarantees that $y$ can be chosen angelically regardless of $x$. We also allow sub-contexts to be connected with dependent conjunction ($\Gamma_1 \ctcomma \Gamma_2$) and disjoint conjunction ($\Gamma_1 \ctstar \Gamma_2$). This bunched modal type system enables us to track complex data-flow dependencies as tree-structured contexts and support framing-based local reasoning.


\paragraph{Contributions.}
We summarize the main technical contributions as follows.
\begin{itemize}[leftmargin=*]
\item We interpret context in functional languages as a resource, and introduce a resource algebra for compositional decomposition and combination of contextual pieces.
\item We define a unified modal \emph{context logic} that supports dependent and disjoint mixing of overapproximation/demonic and underapproximation/angelic modalities.
\item We design \emph{context types}, a bunched modal type system interpreted in our context logic, to support both correctness and incorrectness reasoning for even deterministic functional programs.
\item We formulized our core language and type system in Rocq, and provide corresponding helper tactics to an interactive proof assistant to reason about context types.
\end{itemize}

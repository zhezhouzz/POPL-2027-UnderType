\section{Introduction}\label{sec:intro}

Type-based reasoning has become a standard approach to functional program verification.
Refinement types are a prominent example: they support overapproximate safety reasoning by embedding logical safety properties directly into types.
Under a typing judgment \(\vdash e : \tau\), every result to which \(e\) may reduce should lie in the set described by \(\tau\), so \(\tau\) is an overapproximation of \(e\)'s behavior.
This is a \emph{may}-style reading, since not every behavior allowed by \(\tau\) needs to be realized by \(e\). Moreover, the type context is typically interpreted in a universal style: \emph{for any} allowed context, the typing judgment should hold.
The dual reading treats \(\tau\) as an underapproximate subset of behaviors that \(e\) must realize, and interprets the context in an existential style, which facilitates reachability and completeness reasoning.
Coverage types~\cite{CoverageType} demonstrate the practical value of this view by verifying coverage completeness of test input generators: expected good behaviors are not merely permitted, but guaranteed to be attainable.
More broadly, must-style reasoning has been shown to reduce false positives in bug detection~\cite{OHP19,Ill-types} and to provide reachability guarantees.

Unifying over- and underapproximate reasoning within one semantic framework is not just a theoretically elegant proposal; it has been a recurring goal in verification theory and tooling.
Classic compositional may/must analysis already illustrates the potential synergy: may-style reasoning prunes infeasible regions, while must-style evidence narrows witnesses and localizes specification or implementation defects~\cite{CompositionalMayMust}.
In current type-based verification practice, however, overapproximate and underapproximate reasoning are still often separated at the judgment level.
For example, refinement-typing workflows often rely on external mechanisms such as symbolic execution to search for concrete counterexamples to refinement obligations, rather than integrating underapproximate reasoning directly into typing judgments~\cite{RefinementTypeSymbolicExecution}.
This split leaves overapproximate typing judgments and underapproximate witnesses at different methodological layers, limiting the modularity and reuse that type-system-native reasoning would otherwise provide.
Our goal is therefore not to bolt must-style analysis onto refinement typing, but to internalize both modalities uniformly in typing rules for higher-order functional programs.

Building a type-theoretic foundation for over- and underapproximate reasoning in functional languages is non-trivial.
Standard refinement typing can be read as ``overapproximated parameters'' (interpreted as \emph{for any} inputs) plus an ``overapproximated return'' (interpreted as an upper bound on application results).
Coverage types use ``overapproximated parameters'' with an ``underapproximated return'' to express coverage-completeness guarantees of test generators for any input.
However, these systems usually treat each curried premise as uniformly ``purely over'' or ``purely under'' in isolation, which loses the ability to mix over- and underapproximation modalities in arbitrary ways.
For instance, consider a curried refinement type whose argument positions interchange over- and underapproximate qualifiers:
\begin{align*}
    \vdash \Code{existsElem} : n{:}\ort{\Nat}{\top} \sarr \urt{\List[\Nat]}{\top} \sarr \urt{\Bool}{\vnu = \true} 
\end{align*}\noindent
The first parameter is overapproximated (colored in pink), while the second parameter and return type are underapproximated (colored in blue).
This specification says: ``\textcolor{magenta}{\emph{for any}} element $n$, if the second parameter \textcolor{CobaltBlue}{\emph{covers}} all possible lists, a desired \Code{existsElem} function \textcolor{CobaltBlue}{\emph{must}} be able to return $\true$.'' 
This function type mixes modality patterns along the spine rather than collapsing either side of the arrow to a uniform over- or underapproximate reading.
Hence the design goal goes beyond attaching one extra modality tag to overapproximation: a unified system must
\emph{compositionally reconcile} arbitrary modality regimes throughout higher-order pipelines.

When we consider fine-grained over- and underapproximate modalities for a multi-parameter function, one challenge is dependency between parameters.
Consider the following potential client of \Code{findElem}:
\begin{minted}[xleftmargin=5pt, numbersep=4pt, linenos = true, fontsize = \footnotesize, escapeinside=??]{OCaml}
    let x = nat_gen () in
    let l = nat_list_gen () in
    let l' = if mem l x then [] else l in
    findElem x l'
\end{minted}
$x$ is a random element and $l$ is a random list.
More interestingly, we construct a new list $l'$ that contains only lists that do not contain $x$.
This is a ``wrong case'' where \Code{findElem} can never return $\true$.
At first glance, $x$ and $l'$ seem to satisfy the parameter types of \Code{findElem}: $x$ is always a natural number, and $l'$ may look as if it can still cover all lists.
But there is a hidden dependency: if we require $l'$ to cover, say, $[0;1]$, then $x$ can no longer be $0$ or $1$.
In contrast, \Code{findElem} assumes the two parameter choices are independent.
In general, fine-grained over/underapproximate reasoning requires a typing language expressive enough to describe parameter dependencies, and a type system that can track data-flow dependencies modularly.

The situation becomes more complicated for higher-order functions, where a function itself is an input.
An input function may produce results with over- or underapproximate modality, but what is the relation between repeated results of the same function?
Consider:
\begin{align*}
    &\vdash \lambda g \doteq f.(f\;() == f\;()) : f{:}(\Unit\sarr\urt{\Bool}{\top})\sarr\urt{\Bool}{\vnu = \false}
\end{align*}\noindent
where we expect $f$ to be able to produce different results across two applications.
Note that ``producing two different values one by one'' and ``producing two different values nondeterministically'' are quite different.
For example, for
\begin{minted}[xleftmargin=5pt, numbersep=4pt, linenos = true, fontsize = \footnotesize, escapeinside=??]{OCaml}
    let x = bool_gen () in let f = ?$\lambda ().$?x in (* body *)
\end{minted}
the function $f$ can cover all boolean values nondeterministically, but in the let-body above it can only generate the same value on both calls.
Thus, the application \(g\;f\) can only evaluate to $\true$, which is not what we expect.
Whether a higher-order function is ``duplicatable'' is therefore another feature that our type language must handle.

This paper develops \emph{resolution types}, a resource-modal refinement type system that keeps over- and underapproximation as first-class modalities and allows fine-grained mixing of the two in arbitrary ways.
For a nondeterministic program, demonic execution corresponds to overapproximation (all possible executions must be as expected), while angelic execution corresponds to underapproximation (there exists some execution as expected).
For example, given two reductions $\mstep{e}{v_1}$ and $\mstep{e}{v_2}$, a demonic resolution considers all combinations of these reductions (to $v_1$, to $v_2$, or both), while an angelic resolution selects as needed.
Moreover, the resolving power that enables demonic or angelic choices is treated as a resource in the type system.
At the extreme, for a singleton-valued term (e.g., $\mstep{1+1}{2}$), angelic and demonic resolutions collapse, since there is no space for them to differ.
With this resource-based view, we build an underlying resource algebra and resource logic that provide a Kripke semantics for over- and underapproximate modalities.

This resource-based view also addresses both dependency tracking and function duplicability.
In general, over- and underapproximate modalities can be related in two ways: dependent or independent, which naturally recalls classical bunched typing~\cite{BunchedType} and other resource-based disjointness reasoning approaches.
For example, our type system provides a \emph{magic wand} ($\ctwand$), in addition to the normal arrow ($\ctsarr$), to express independence requirements between function parameters:
\begin{align*}
    \vdash \Code{existsElem} : n{:}\ort{\Nat}{\top} \ctwand \urt{\List[\Nat]}{\top} \ctwand \urt{\Bool}{\vnu = \true} 
\end{align*}\noindent
With bunched typing, we can track complex data-flow dependencies in a tree-structured context and apply framing rules for local reasoning.
To express function duplicability, we also introduce a \emph{persistent} modality $\ctpers$, meaning the function is independent of demonic or angelic decisions made by the context (i.e., the contextual resource is singleton).
The function then makes decisions solely from the nondeterminism encapsulated in its body, and is therefore allowed to produce different results across multiple applications.

\paragraph{Contributions.}
We summarize the main technical contributions as follows.
\begin{itemize}[leftmargin=*]
\item We define a \emph{resolution algebra} of resolution resources and a modal \emph{resolution logic} whose connectives interpret approximation operators, dependence, and persistence.
\item We present \emph{resolution types} for functional programs, allowing users to mix over- and underapproximate modalities in arbitrary ways within types. We give resolution types a compositional denotational semantics in resolution logic.
\item We design declarative typing and subtyping rules sufficient to prove semantic soundness of resolution types, and evaluate the design through case studies that combine over- and underapproximate reasoning.
\end{itemize}
